# Local variable name shadows variable in outer scope
**ID:** `JS-0123` | **Link:** [DeepSource](https://deepsource.com/directory/javascript/issues/JS-0123)

![Minor](https://img.shields.io/badge/severity-minor-yellow)![Bug Risk](https://img.shields.io/badge/type-bug_risk-green)

Two variables can have the same name if they're declared in different scopes. In the example below, the parameter `x` is said to "shadow" the variable `x` declared above it. The outer `x` can no longer be accessed inside the `sum` function.


```javascript
const x = 1
function add(x, y) {
    return x + y
}
```
While shadowing does not cause any problems most of the time, it does make the code harder to read and understand. We highly recommend against shadowing. However, if you want to shadow some variable name and don't want DeepSource to flag it, add a [skipcq comment](https://docs.deepsource.com/docs/issues-ignore-rules) alongside an explanation:


```javascript
const x = 1
function add(x, y) { // skipcq: JS-0123 - `x` can be safely shadowed
    return x + y
}
```
If you want to disable this issue project-wide, you can add it to the list of disabled issues in the project dashboard.


## Bad Practice

```javascript
const file = "data.txt"

function readFile(file) {
    // The parameter `file` shadows the toplevel variable `file`.
    if (fs.existsSync(file)) {
        return fs.readFileSync(file)
    }
    return null
}
```

## Recommended

```javascript
// Prefer variable names that are distinct and convey as much
// meaning as possible.
const dataFile = "data.txt"

function readFile(filePath) {
    if (fs.existsSync(filePath)) {
        return fs.readFileSync(filePath)
    }
    return null
}
```
Alternatively:


```javascript
const file = "data.txt"

function readFile(file) { // skipcq: JS-0123 - Shadowing is safe here
    // ...
}
```
