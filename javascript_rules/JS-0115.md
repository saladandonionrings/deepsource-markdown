# Prefer the usage of regular expression literals over the `RegExp` constructor
**ID:** `JS-0115` | **Lien:** [DeepSource](https://deepsource.com/directory/javascript/issues/JS-0115)

![Minor](https://img.shields.io/badge/severity-minor-yellow) ![Anti-pattern](https://img.shields.io/badge/type-anti_pattern-purple)

When a regular expression is known in advance, it is considered a best practice to avoid the string literal notation on top of the regular expression notation, and use regular expression literals instead of the constructor function.


```javascript
// bad practice:
const number  = new RegExp("^\d\.$");
// recommended:
const number_ = /^\d\.$/;
```
Regex literals are easier to read and do not require a call to the `RegExp` constructor at runtime.


## Bad Practice

```javascript
new RegExp("abc");

new RegExp("abc", "u");

RegExp("abc");

RegExp("abc", "u");

new RegExp("\d\d\.\d\d\.\d\d\d\d");

RegExp(`^\d\.$`);

new RegExp(String.raw`^\d\.$`);
```

## Recommended

```javascript
/abc/;

/abc/u;

/\d\d\.\d\d\.\d\d\d\d/;

/^\d\.$/;

// RegExp constructor is allowed for dynamically generated regular expressions

new RegExp(pattern);

RegExp("abc", flags);

new RegExp(prefix + "abc");

RegExp(`${prefix}abc`);

new RegExp(String.raw`^\d\. ${sufix}`);
```
