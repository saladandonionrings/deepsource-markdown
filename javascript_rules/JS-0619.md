# Avoid using `v-if` on the same element as `v-for`
**ID:** `JS-0619` | **Link:** [DeepSource](https://deepsource.com/directory/javascript/issues/JS-0619)

![Minor](https://img.shields.io/badge/severity-minor-yellow)![Anti-pattern](https://img.shields.io/badge/type-anti_pattern-purple)

It is recommended to avoid the use of `v-for` directives with `v-if` directives on the same element because when vue processes directives, `v-if` has a higher priority than `v-for`, so the `v-if` condition will not have access to variables from the scope of the `v-for`.

There are two common cases where this can be tempting:

* To filter items in a list (e.g.`v-for="user in users" v-if="user.isActive"`).
In these cases, replace users with a new computed property that returns your filtered list (e.g.`activeUsers`).* To avoid rendering a list if it should be hidden (e.g.`v-for="user in users" v-if="shouldShowUsers"`).
In these cases, move the`v-if`to a container element (e.g.`ul`,`ol`).
Consider this example


```javascript
<ul>
  <li
    v-for="city in cities"
    v-if="city.isServiceable"
    :key="city.id"
  >
    {{ city.name }}
  </li>
</ul>
```
Here vue will throw error as `v-if` is evaluated first and the iteration variable `city` will not exist at this moment.


## Bad Practice

```javascript
<template>
    <TodoItem
      v-if="complete"
      v-for="todo in todos"
      :todo="todo"
    /><!-- ↑ In this case, the `v-if` should be written on the wrapper element. -->
    <TodoItem
      v-for="todo in todos"
      v-if="todo.shown"
      :todo="todo"
    /><!-- ↑ In this case, the `v-for` list variable should be replace with a computed property that returns your filtered list. -->
  </template>
```

## Recommended

```javascript
<template>
<ul v-if="complete">
      <TodoItem
        v-for="todo in todos"
        :todo="todo"
      />
</ul>
<TodoItem
    v-for="todo in shownTodos"
    :todo="todo"
/>
</template>
```
