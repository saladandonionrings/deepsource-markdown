# Found usage of exported name as property of default import
**ID:** `JS-P1003` | **Link:** [DeepSource](https://deepsource.com/directory/javascript/issues/JS-P1003)

![Minor](https://img.shields.io/badge/severity-minor-yellow)![Performance](https://img.shields.io/badge/type-performance-white)

When there are named and default exports present and when the named export is being used as property or member of default import, this thus affects the performance of the build tools while performing optimization.

We can access these named exports in two ways.

* using named import i.e using `import { sum } from './calc.js'* using the member or property of the default import, i.e., using this

```javascript
import calc from './calc.js'

console.log(calc.sum(2, 3));
```
Here the first one is recommended as it makes the code more readable and easy to refactor. And the primary reason to prefer the 1st one is it improves the performance. That is, as here can notice when importing through default imports, all the named exports are also available as a member that includes exports that might not be needed. So this adds unnecessary time and space to have them during the execution of the code.


## Bad Practice

```javascript
// <!-- calc.js -->

export default function calc(){
    return "you got the calculator"
}

export function sum(a, b){ return a + b; }
```

```javascript
// <!-- index.js -->

import calc from './calc.js'

console.log(calc.sum(1, 2))
```

## Recommended

```javascript
// <!-- calc.js -->

export default function calc(){
    return "you got the calculator"
}

export function sum(a, b){ return a + b; }
```

```javascript
// <!-- index.js -->

import { sum } from './calc.js'

console.log(sum(1, 2))
```
