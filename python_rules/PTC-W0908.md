# Use of `unique_for` constrant found
**ID:** `PTC-W0908` | **Lien:** [DeepSource](https://deepsource.com/directory/python/issues/PTC-W0908)

![Minor](https://img.shields.io/badge/severity-minor-yellow) ![Bug Risk](https://img.shields.io/badge/type-bug_risk-green)

`unique_for_date/month/year` is very brittle and can be a source for unexpected behaviour.

The unique_for_date/month/year constraint is enforced at the Django admin-form level but not at the database level. Checking will not occur if `Model.save()` is called without first calling `Model.validate_unique()` . Only the date portion of the field will be considered, even when validating a `DateTimeField` .

Refer to the [docs](https://docs.djangoproject.com/en/3.1/ref/models/fields/#unique) to read more about `unique_for` constraint.


## Bad practice

```python
from django.db import models

class MyModel(models.Model):
    date = models.DateField()
    somefield = models.CharField(unique_for_date='date')
```

## Recommended

```python
from django.db import models

class MyModel(models.Model):
    date = models.DateField()
    somefield = models.CharField()

    def save(self, *args, **kwargs):
        if self.objects.filter(date=self.date, somefield=self.somefield).exists():
            raise ValidationError({'somefield': 'Value'})
        return super().save(*args, **kwargs)
```
